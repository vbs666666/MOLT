# 需求文档

## 1. 应用概述

- **应用名称**：MOLT
- **应用描述**：MOLT 是一款面向身份过渡期人群的 Web 应用，帮助用户看见自己当下所处的位置与状态，并与同样处于过渡期的人建立连接。应用通过分路径的对话式引导、自我镜像确认与可视化地图，让用户在「换壳」过程中感受到被看见、被陪伴。

---

## 2. 用户与使用场景

### 2.1 目标用户

- 正在经历职业转型、身份转变或人生阶段切换的人
- 已完成过渡、希望以经验帮助他人的人

### 2.2 核心使用场景

- 用户在迷茫或挣扎时，通过对话式引导梳理自身状态
- 用户完成自我描述后，在地图上看见与自己相似的他人轨迹
- 走过过渡期的用户建档留存，为后来者提供参照

---

## 3. 页面结构与核心功能

### 3.1 页面总览

```
MOLT
├── Landing 页
├── Onboarding 页
├── 对话页
│   ├── Path A（挣扎路径）
│   ├── Path B（挣脱路径）
│   └── Path C（回望路径）
├── 镜子确认页
├── 结果页（Path A / B）
├── 灯塔建档页（Path C）
└── 地图页
```

### 3.2 Landing 页

**视觉规格**
- 背景色：#0a0a0a
- 主色：荧光绿 #7CFF6B
- 字体：等宽英文 JetBrains Mono + 无衬线中文黑体
- 全屏单焦点，无导航栏

**核心内容**
- 大字宣言（居中展示）：「你不是被时代淘汰的人。你是正在换壳的人。」
- 主按钮：「开始脱壳」
  - 点击后跳转至 Onboarding 页

### 3.3 Onboarding 页

**布局**
- 三张卡片横向或纵向排列，每次聚焦一张
- 无导航栏，一屏一焦点

**卡片定义**

| 卡片 | 标题 | 描述文字 | 按钮文案 | 跳转路径 |
|------|------|----------|----------|----------|
| A | 挣扎 | 还在原地，动弹不得 | 进入 Path A | 对话页 Path A |
| B | 挣脱 | 已经出发，但不确定方向 | 进入 Path B | 对话页 Path B |
| C | 回望 | 走过来了，想看看能不能帮到别人 | 进入 Path C | 对话页 Path C |

- 每张卡片包含：标题、描述文字、跳转按钮
- 用户点击卡片按钮后，进入对应路径的对话页

### 3.4 对话页

**通用交互规则**
- 一次仅展示一个问题，用户作答后方可进入下一题
- 用户回答以文本输入框收集
- 所有路径的回答均存储至后端，用于镜子确认页复述
- 每个路径问题数量建议 3～5 题（具体题目由产品内容团队定义，PRD 仅约定结构与语气规范）

**Path A — 挣扎**
- 语气：慢、温、情绪承接
- 问题风格：先共情，再轻轻引导用户说出卡住的位置
- 示例问题方向：「最近让你觉得最重的一件事是什么？」

**Path B — 挣脱**
- 语气：快、平等、务实
- 问题风格：直接切入现状与资源盘点
- 示例问题方向：「你现在手里有什么？你想去哪里？」

**Path C — 回望**
- 语气：郑重、受邀、采访感
- 问题风格：以采访者视角邀请用户回顾轨迹
- 示例问题方向：「你是从哪个时刻开始感觉到转变的？」

**进度指示**
- 页面顶部或底部展示当前问题序号（如「2 / 4」）
- 提供「上一题」返回能力

**完成后跳转**
- Path A / B → 镜子确认页
- Path C → 镜子确认页

### 3.5 镜子确认页

**核心功能**
- 将用户在对话页中的回答，以用户自己的原话或近似原话进行结构化复述
- 复述内容以卡片或段落形式呈现，视觉上有明显的「镜像感」

**交互按钮**
- 「是的，继续」：确认复述内容，跳转至对应结果页
  - Path A / B → 结果页
  - Path C → 灯塔建档页
- 「我来修正」：返回对话页，允许用户修改回答

### 3.6 结果页（Path A / B）

**模块一：置换压力指数**
- 根据用户回答计算并展示压力指数区间（如低 / 中 / 高）
- 以可视化区间条或数值区间形式呈现
- 附简短说明文字解释该区间含义

**模块二：能力重估**
- 基于用户回答，提炼并展示用户当前具备的能力标签或描述
- 以标签列表或卡片形式呈现

**模块三：市场信号与行动建议**
- 展示与用户当前状态相关的市场信号（方向性描述）
- 给出 1～3 条具体可执行的行动建议

**底部引导**
- 提供入口跳转至地图页，查看其他用户轨迹

### 3.7 灯塔建档页（Path C）

**模块一：轨迹摘要确认**
- 展示从对话页收集的内容摘要，供用户最终确认
- 提供「修改」入口返回对话页

**模块二：公开意愿设置**
- 单选项：
  - 公开展示在地图上（其他用户可见）
  - 仅自己可见
- 默认选中「公开展示」

**模块三：完成建档**
- 按钮：「完成建档」
- 点击后将档案存储至后端，并跳转至地图页
- 若用户选择公开，其节点将出现在地图中

### 3.8 地图页

**可视化组件**
- 使用 D3.js 力导向图渲染
- 每个节点代表一位用户的轨迹（Path C 建档用户，且设置为公开）
- 节点颜色 / 大小可根据路径类型或状态阶段区分（具体视觉规则由设计定义）
- 节点可点击，展示该用户的轨迹摘要（脱敏处理）

**发送信号功能**
- 当前用户可向某节点「发送信号」
- 信号为简短文本（字数上限建议 50 字）
- 发送后在地图上产生视觉反馈（如连线或粒子动效）
- 信号内容存储至后端，关联发送方与接收方节点

**当前用户节点**
- 当前用户在地图上有自己的节点（若已完成任意路径）
- 高亮区分当前用户节点与他人节点

## 4. 业务规则与逻辑

### 4.1 路径分流规则
- 用户在 Onboarding 页选择卡片后，全程跟随对应路径（A / B / C）
- 路径一经选择，当次会话内不可中途切换
- 用户可重新从 Landing 页进入，开启新一次流程

### 4.2 数据存储规则
- 对话页的每条用户回答实时存储至 PostgreSQL 数据库
- 镜子确认页的复述内容基于已存储回答在前端生成，不单独存储
- 结果页的压力指数、能力标签、行动建议基于回答在服务端计算后返回并存储
- 灯塔建档页的档案（含公开意愿）存储至数据库，公开档案可被地图页查询
- 发送信号记录存储至数据库，包含：发送方用户 ID、接收方用户 ID、信号内容、发送时间

### 4.3 地图节点可见性规则
- 仅 Path C 完成建档且设置为「公开」的用户节点在地图上可见
- Path A / B 用户在地图上有节点，但默认不展示轨迹摘要，仅作为匿名节点存在（可根据产品策略调整）
- 用户可随时在个人设置中修改公开意愿（本期不实现，列入后续规划）

### 4.4 用户标识规则
- 本期不引入注册 / 登录体系
- 使用浏览器本地存储（localStorage）生成并持久化匿名用户 ID
- 匿名 ID 用于关联用户的对话记录、结果数据与地图节点

## 5. 异常与边界情况

| 场景 | 处理方式 |
|------|----------|
| 用户在对话页未填写回答直接点击下一题 | 输入框高亮提示，阻止跳转，提示「请先填写回答」 |
| 用户在镜子确认页点击「我来修正」后再次返回 | 保留上次已填写的回答内容，不清空 |
| 地图页节点数量为零（冷启动） | 展示空状态提示文案，如「还没有人留下轨迹，你会是第一个吗？」 |
| 发送信号内容超出字数上限 | 实时计数提示，超出后禁止提交 |
| 用户本地存储被清除导致匿名 ID 丢失 | 重新生成新匿名 ID，历史数据不可恢复，不做额外提示 |
| 网络请求失败（对话提交 / 建档 / 发送信号） | 展示通用错误提示，提供重试按钮 |

## 6. 验收标准

- Landing 页在 #0a0a0a 背景下正确渲染宣言文字与「开始脱壳」按钮，点击后跳转 Onboarding 页
- Onboarding 页三张卡片内容完整，点击各卡片按钮后分别进入对应路径对话页
- 对话页每次仅展示一个问题，用户作答后方可进入下一题，进度指示正确更新
- Path A / B / C 三条路径语气风格符合定义，问题顺序正确
- 镜子确认页正确复述用户回答内容，「是的，继续」与「我来修正」按钮行为符合预期
- 结果页（Path A / B）正确展示压力指数区间、能力标签与行动建议三个模块
- 灯塔建档页（Path C）正确展示轨迹摘要，公开意愿设置可操作，「完成建档」后数据写入数据库并跳转地图页
- 地图页 D3.js 力导向图正常渲染，节点可点击展示摘要，发送信号功能可用且字数限制生效
- 全程无导航栏，每屏聚焦单一内容，UI 色彩与字体规格符合设计定义
- 匿名用户 ID 在同一浏览器内持久化，刷新页面后不丢失
